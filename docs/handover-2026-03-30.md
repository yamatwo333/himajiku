# 引き継ぎメモ (2026-03-30)

## 目的

`himajiku` の現行実装を、次に触る人が短時間で追えるように整理したメモです。コードの挙動変更は含まず、2026-03-30 時点の `main` を読んでまとめています。

## 全体像

- Next.js App Router のモバイル前提 Web アプリ。
- 認証は LINE Login -> Supabase Auth セッション化。
- 「ヒマ」データは `availability` にユーザー単位で保存され、グループ単位では持っていません。
- グループは `groups` / `group_members` で管理し、閲覧時に所属メンバーの `availability` を絞り込みます。
- LINE 通知はグループごとの `notify_threshold` を超えた時間帯だけ送信します。

## 画面と責務

### ログイン

- `/login`
  - LINE ログイン導線のみ。
  - `redirect` クエリがあれば `/auth/line` に引き継ぎます。
- `/auth/line`
  - LINE OAuth の `state` を cookie に保存して認可画面へリダイレクトします。
- `/auth/line/callback`
  - code exchange -> LINE profile 取得 -> Supabase Admin でユーザー作成 or 既存利用 -> magic link 生成 -> `verifyOtp` でセッション cookie 発行、という流れです。
  - `login_redirect` cookie があれば `/calendar` 以外にも戻せます。

### 認証ガード

- `src/middleware.ts`
  - 未ログインで保護ページに行くと `/login?redirect=...` に飛ばします。
  - `/login` にログイン済みで来た場合は `redirect` か `/calendar` に戻します。
  - `/api`, `/auth`, `/join`, `/login` は公開扱いです。

### カレンダー

- `/calendar`
  - 最初に `/api/groups/mine` で所属グループを取得。
  - 選択グループに応じて `/api/availability/month` を叩き、カレンダー表示を作ります。
  - `calendarMonth` と `selectedGroupId` を `sessionStorage` に保存しているため、詳細画面から戻っても月とグループ選択が維持されます。
- `/calendar/[date]`
  - 日別詳細画面。
  - `/api/availability/[date]` を叩き、当日のヒマ状況と自分の登録内容を取得します。
  - `group` クエリは表示対象メンバーの絞り込み用で、保存先のレコード自体はグループ非依存です。
  - 画面文言にもある通り、1 回の保存は参加中の全グループに実質反映されます。
- `/calendar/bulk`
  - 1 か月分をまとめて登録。
  - 保存は `/api/availability/bulk` に集約されています。

### グループ

- `/groups`
  - 一覧取得は `/api/groups/mine`。
  - 新規作成は `/api/groups`、招待コード参加は `/api/groups/join`。
- `/groups/[id]`
  - 詳細、メンバー一覧、招待リンク/コード、管理者変更、脱退、削除をまとめた画面です。
  - グループ名と通知しきい値は `useEffect` の自動保存で `/api/groups/[id]/settings` を呼びます。
  - LINE 連携コードは `/api/groups/[id]/line-link` で発行・解除します。
  - 管理者変更は `/api/groups/[id]/transfer`、脱退は `/api/groups/[id]/leave`、削除は `/api/groups/[id]` の `DELETE`。
- `/join`
  - 招待リンク経由の参加専用画面。
  - 認証直後でセッションが張られるまでを見込んで、401 時は最大 3 回リトライしてから `/login?redirect=...` に送ります。

### マイページ

- `/profile`
  - `profiles` から表示名とアイコンを読むだけの軽い画面です。

## API とデータの流れ

### availability 系

- `POST /api/availability`
  - 1 日分の保存/削除。
  - 保存後に所属グループ全件へ `/api/notify` を非同期発火します。
  - ついでに 3 か月より古い `availability` を削除します。
- `GET /api/availability/[date]`
  - 日別一覧。
  - `group` 指定時は、そのグループのメンバーだけに絞り込みます。
- `GET /api/availability/month`
  - 月表示用の範囲取得。
- `POST /api/availability/bulk`
  - 複数日まとめ保存。
  - こちらも所属グループ全件へ通知チェックを投げます。

### groups 系

- `POST /api/groups`
  - グループ作成後、作成者を `group_members` に追加します。
- `POST /api/groups/join`
  - 招待コードで参加。
- `GET /api/groups/mine`
  - 所属グループとメンバー数を返します。
- `GET /api/groups/[id]`
  - メンバー確認後、グループ情報とプロフィール付きメンバー一覧を返します。
- `PUT /api/groups/[id]/settings`
  - グループ名と `notify_threshold` を更新します。
- `POST /api/groups/[id]/leave`
  - 管理者なら最古参メンバーへ所有権移譲、他に誰もいなければグループ削除。
  - `availability` はグローバルデータなので消しません。
- `POST /api/groups/[id]/transfer`
  - 明示的な管理者変更。

### LINE 通知系

- `POST /api/groups/[id]/line-link`
  - 6 文字コードを発行し、10 分だけ有効にします。
- `DELETE /api/groups/[id]/line-link`
  - LINE グループ連携解除。
- `POST /api/line/webhook`
  - LINE Messaging API webhook。
  - `join` イベントで案内文を返し、`leave` で DB の `line_group_id` を外します。
  - `連携 XXXXXX` メッセージでシェアヒマ側グループとの紐付けを確定します。
- `POST /api/notify`
  - 指定日・指定グループについて、同じ時間帯に `notify_threshold` 人以上いるかを判定。
  - LINE 設定がある場合だけ push 送信します。
  - 通知メッセージの詳細リンクは `NEXT_PUBLIC_APP_URL`、未設定時は `https://sharehima.vercel.app` を使います。

## DB / Supabase

### 主テーブル

- `profiles`
  - `auth.users` と 1:1。
  - 新規ユーザー時の trigger で自動作成される想定です。
- `availability`
  - `user_id + date` で一意。
  - `time_slots` は text array。
- `groups`
  - 招待コード、通知しきい値、LINE 連携状態を保持します。
- `group_members`
  - グループ所属管理。

### migrations の流れ

- `001_initial_schema.sql`
  - `profiles` / `availability` と trigger 群を作成。
- `002_groups.sql`
  - `groups` / `group_members` を追加。
- `003_fix_group_rls.sql`
  - グループ作成直後に作成者が見えなくなる問題を避ける方向で RLS を調整。
- `004_line_bot_groups.sql`
  - いったん `line_bot_groups` を追加。
- `005_group_line_link_code.sql`
  - `groups` に `link_code` / `link_code_expires_at` を追加し、`line_bot_groups` は廃止。
- `006_transfer_owner.sql`
  - 実質 no-op コメントのみ。

### 実運用上の注意

- ルートハンドラの多くは、認証確認のあと `SUPABASE_SERVICE_ROLE_KEY` を使う Admin client で読み書きしています。
- そのため、最終的な権限制御は RLS だけでなく route 側条件分岐にも依存しています。
- `ensureProfile` が複数 API から呼ばれており、auth trigger が失敗しても最低限のプロフィールを補完します。

## 既知課題 / 引き継ぎ時の注意

1. Next.js 16 系なのに認証ガードがまだ `src/middleware.ts` のままです。`proxy.ts` への移行候補です。
2. `src/lib/mock-data.ts` は現状どこからも参照されていません。削除候補ですが、デザイン確認用に残した可能性はあります。
3. `/api/groups/[id]/settings` は detail 画面の自動保存から現役で叩かれています。未使用 API ではありません。
4. `https://sharehima.vercel.app` が `src/app/layout.tsx` と `src/app/api/notify/route.ts` に残っており、デプロイ先 URL 変更時は差し替え漏れに注意が必要です。
5. `availability` がグループ非依存なので、「あるグループを抜けたらそのグループ向けのヒマも消える」とはなりません。仕様として問題ないかは将来再確認余地があります。
